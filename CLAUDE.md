# edp-cluster-add-ons

GitOps Helm add-on catalog for KubeRocketCI — a curated set of pre-configured Helm wrapper charts deployed via Argo CD's App-of-Apps pattern. No compiled code; all changes are declarative YAML/Helm.

## Directory Layout

```
argo-cd/                         # Standalone Helm chart for bootstrapping Argo CD itself
                                 # (separate from the add-ons catalog; applied first)
clusters/
  core/
    bootstrap-addons.yaml        # Applied once manually to seed the Argo CD App-of-Apps
    apps/                        # App-of-Apps chart: one template per add-on, enable flags only
      Chart.yaml
      values.yaml                # enable: true/false + namespace per add-on
      templates/
        cert-manager.yaml        # Each file is an ArgoCD Application that points to the add-on
        ingress-nginx.yaml       # ...
    addons/                      # Individual add-on wrapper charts
      cert-manager/              # Each is a thin Helm chart wrapping an upstream dependency
        Chart.yaml               # version = upstream chart version; declares the dependency
        values.yaml              # All add-on config goes here, keyed by dependency chart name
        README.md                # helm-docs generated
      ingress-nginx/
      external-secrets/
      ...
  prod/
    bootstrap-addons.yaml        # Points prod Argo CD at clusters/prod/apps (remote cluster)
    apps/
      templates/                 # Only overrides/extra apps for prod (e.g. vault-remote-rbac)
    addons/
      vault-remote-rbac/         # Add-ons that exist only in prod
docs/                            # Add-on and cluster setup guides
hack/                            # Python scripts: update_readme.py, check_dependencies.py
```

## App-of-Apps Pattern

`clusters/core/bootstrap-addons.yaml` is the single Argo CD Application applied manually. It points to `clusters/core/apps`, which is itself a Helm chart. Each template in `clusters/core/apps/templates/` generates an Argo CD `Application` resource pointing at the corresponding directory under `clusters/core/addons/`. An add-on is deployed when `enable: true` is set in `clusters/core/apps/values.yaml`.

`clusters/prod/` mirrors the same structure but targets an external cluster (`destination.server` is a remote URL, not `kubernetes.default.svc`). Its `apps/templates/` only contains prod-specific additions; most add-ons re-use the core definitions.

## Add-on Structure Convention

Each add-on under `clusters/<cluster>/addons/<name>/` is a **wrapper chart**:

- `Chart.yaml` — declares the upstream chart as a `dependencies` entry; `version`/`appVersion` both track the upstream version exactly.
- `values.yaml` — all configuration lives here, using the **dependency chart's name** as the top-level key so values are passed through to the subchart. No custom config belongs in `apps/values.yaml`.
- `templates/` — only needed for custom Kubernetes resources beyond what the upstream chart provides (e.g. ExternalSecret, AppProject, Ingress).
- `README.md` — generated by `helm-docs`; use `README.md.gotmpl` to customize.

App-of-Apps template (`clusters/core/apps/templates/<name>.yaml`) pattern:

- Guard: `{{- if and (index .Values "addon-name") (index .Values "addon-name" "enable") -}}`
- Use `index .Values "addon-name"` syntax (not `.Values.addon-name`) because add-on names contain dashes.
- `apps/values.yaml` for the add-on must only have `enable`, `namespace`, and optionally `createNamespace`.

If the upstream chart needs a new Helm repo, add it to the `chart-repos` list in `ct.yaml`.

## Lint / Validation Commands

```bash
# Lint only changed charts (what CI runs on PRs)
make ci-lint-chart

# Lint all charts
make lint-all-charts

# Lint a single chart
make lint-chart CHART=cert-manager

# Regenerate per-add-on README.md files (required before committing)
make helm-docs

# Validate README files are up-to-date (CI checks this)
make validate-docs

# Regenerate root README.md table of add-ons
make update-readme

# Check whether upstream chart versions have newer releases
make check-dependencies               # requires virtualenv (.venv)
make check-dependencies ARGS="--workers 5"

# Full install test (lint → kind cluster → ct install)
make test-charts-full
make kind-delete                      # clean up afterwards
```

CI runs `make ci-lint-chart` and `make update-readme` on every PR touching `clusters/**`. Chart install tests in kind are commented out in CI but available locally via `make test-charts-full`.

Lint rules are defined in `lint-conf.yaml` (yamllint ruleset); chart-testing config is in `ct.yaml` (chart dirs scoped to `clusters/core/addons`).

## Adding a New Add-on

1. `mkdir -p clusters/core/addons/<name>` — create `Chart.yaml`, `values.yaml`, `README.md` (see `docs/add-new-addon.md` for templates).
2. Add the upstream chart repo to `chart-repos` in `ct.yaml` if not already present.
3. Create `clusters/core/apps/templates/<name>.yaml` following the App-of-Apps template pattern above.
4. Add the enable/namespace entry to `clusters/core/apps/values.yaml` (keep the file sorted alphabetically).
5. Run `make helm-docs` to generate `README.md`, then `make lint-chart CHART=<name>` to validate.
6. Run `make update-readme` to refresh the root `README.md` add-on table.

See `docs/add-new-addon.md` for the full walkthrough and `docs/add-new-cluster.md` for adding a new cluster environment.
